## Incidents

- An `Incident` is created when a monitor goes DOWN (or DEGRADES, if you choose).
- Timeline events live in `IncidentEvent`.
- API routes:
  - `GET /incidents` — list incidents for the authenticated user.
  - `GET /incidents/:incidentId/timeline` — event history.

## Notifications

- Notification channels: email, SMS, Slack (extensible).
- Create channel: `POST /create-channel` with `{ type, config }`.
- List: `GET /get-channels`
- Delete: `DELETE /delete-channel/:id`
- Attach channels to a monitor: `PUT /monitors/:id/channels` with `channelIds: string[]`.

## Status pages

- One public status page per monitor.
- Create: `POST /statuspage/:id` (body: `title`, `description` optional).
- Fetch public data: `GET /statuspage/:monitorId` → returns page info + latest 500 ticks.
- Delete: `DELETE /statuspage/:monitorId`.

## Authentication & sessions

- Sign up: `POST /signup` with `{ email, password, firstName, lastName }`.
- Sign in: `POST /signin` → returns JWT + `sessionId` (1h expiry).
- Include `Authorization: Bearer <token>` for authenticated routes.
- Manage sessions:
  - `GET /sessions` — list active sessions.
  - `POST /logout` — delete current or provided `sessionId`.

## Incident runbook (suggested)

1. **Alert received** → open dashboard → locate monitor.
2. **Inspect checks** → `GET /monitors/:id/checks` for recent failures.
3. **Acknowledge** (if you track manually): update incident status to `ACKNOWLEDGED` (extend API as needed).
4. **Communicate** → post update on the monitor’s status page.
5. **Resolve** → after recovery, mark incident `RESOLVED` (extend API or DB).
